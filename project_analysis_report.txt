                   OS LAB COMPLETE PROJECT ANALYSIS                   
========================================================================
Deep-dive Architectural and Line-by-Line Breakdown
------------------------------------------------------------------------

This report analyzes the custom-built OS Lab Web Server Deployment 
Script and its accompanying test files. This project represents a robust, 
fully automated Ubuntu Server provisioning system combining Apache 
Virtual Hosting, a handcrafted GUI Control Panel, and enterprise-grade 
security hardening.

------------------------------------------------------------------------
 1. GLOBAL ARCHITECTURE SUMMARY
------------------------------------------------------------------------
The complete ecosystem is built from a single monolithic bash script 
(oslab_project (1).sh), which acts as an orchestrator. Upon running:
$ sudo bash oslab_project.sh 

The script performs the following:
1) Installs core dependencies (apache2, ufw, fail2ban, modsecurity-crs, 
   and php).
2) Configures Apache modules and file directory structures.
3) Automatically generates RSA-2048 Self-Signed Certificates.
4) Generates and writes raw HTML and PHP logic to disk directly from 
   Bash heredocs.
5) Hardens the OS network with UFW and the Web Application Layer with 
   ModSecurity and Fail2Ban.
6) Installs a suite of custom CLI tooling and fully functional REST APIs 
   for the interactive Control Panel.


------------------------------------------------------------------------
 2. LINE-BY-LINE BASH SCRIPT BREAKDOWN
------------------------------------------------------------------------
The deployment script spans approximately 2500 lines and is deliberately 
divided into 9 phases plus a Control Panel phase.

>> PHASE 1-3: Dependencies, Modules & Structure (Lines 1 - 143)
* Lines 1-60: Guard checks to ensure the user is 'root'. Initializes CLI 
  colored outputs using terminal escape codes. Uses 'openssl rand' to 
  dynamically generate a secure 16-character base64 password for the 
  admin account, writing it to /root/.oslab_admin_password securely 
  (chmod 600).
* Lines 61-100: Triggers apt-get to install the LAMP-equivalent stack. 
  Note the inclusion of libapache2-mod-security2 (WAF) and 
  modsecurity-crs (OWASP Core Rules). It uses 'sed' to edit php.ini 
  dynamically, pumping upload_max_filesize to 64MB and re-enabling 
  shell_exec and exec which are crucial for the Control Panel's ability 
  to run Bash commands from PHP.
* Lines 101-143: Systematically creates directories for 5 isolated 
  environments: site1.local, site2.local, status.local, admin.local, 
  and panel.local. The command 'chown -R www-data:www-data /var/www/' 
  ensures Apache owns the web root.

>> PHASE 4-5: Certificates & Base Websites (Lines 144 - 571)
* Lines 144-165: A loop calling 'openssl req -x509' to forge self-signed 
  SAN certificates for all .local domains, valid for 365 days.
* Lines 166-335: Writes site1.local/html/index.html (a beautifully 
  styled team portfolio) and site2.local/html/index.html (the OS Lab 
  Project Documentation) natively using bash string redirection.
* Lines 337-432: Crafts status.local/html/index.php. This is a 
  programmatic telemetry dashboard utilizing PHP's shell_exec('free -m'), 
  disk space calculators, and Apache Access Logs parser to visually 
  display system health.
* Lines 475-548: Writes site1.local/waf-test/index.html. This is a 
  frontend Javascript application utilizing the fetch API to actively 
  fire payloads like UNION SELECT, <script>XSS, and Directory Traversals 
  against the server, listening for HTTP 403 Forbidden responses to 
  prove the WAF is functioning.

>> PHASE 6: Core LAN & Virtual Hosting (Lines 572 - 774)
* Lines 577-638: The write_vhost function writes actual Apache 
  <VirtualHost *:80> and <VirtualHost *:443> blocks. It inserts HTTPS 
  Redirect rules, strict security headers, and implements .htpasswd 
  basic authorization if requested.
* Lines 639-761: Generates the LAN Gateway Portal. The script dynamically 
  identifies the server's LAN IP. Devices across the network hit a PHP 
  script (000-portal.conf) that parses /var/www/*/html and dynamically 
  generates a clickable index card for every hosted website utilizing 
  AliasMatch.

>> PHASE 7-8: Firewall, Fail2Ban & WAF (Lines 775 - 997)
* Lines 778-792: Triggers 'ufw default deny incoming', opening only 22 
  (SSH), 80 (HTTP), and 443 (HTTPS).
* Lines 793-834: The script edits /etc/fail2ban/jail.local configuring 
  3 primary jails: sshd, apache-auth, and a custom jail apache-req to 
  ban IPs firing too many requests per minute to Apache.
* Lines 835-997 (Critical Security): Configures ModSecurity WAF. Rather 
  than relying purely on the heavy OWASP CRS, it crafts specific SecRule 
  patterns in /etc/modsecurity/custom_rules/oslab_rules.conf. Phase 1 
  rules intercept URIs. Phase 2 rules parse POST/GET body ARGS looking 
  for bad patterns returning a hard HTTP 403 block.

>> PHASE 9: CLI Automation (Lines 998 - 1194)
* Lines 1003-1121: Writes /usr/local/bin/webmon, an interactive Bash 
  program mapping options (1-9) to standard monitoring commands (like 
  'tail -f *access.log' and 'ufw status').
* Lines 1122-1194: Configures automated Cron Jobs via backup_logs.sh 
  (tars and zips apache logs daily at midnight) and daily_report.sh 
  (compiles a daily overview text file at 6 AM).

>> PHASE 10: The Interactive Control Panel (Lines 1195 - 2400)
* Lines 1201-1225: Elevates the www-data account via sudoers, granting 
  exactly the minimum permissions required to restart Apache, query UFW, 
  ban IPs, and run a2ensite WITHOUT needing a password prompt.
* Lines 1262-1419: Spawns panel.local/html/index.php. Employs a stateful 
  PHP session mechanism overriding access via the securely generated 
  ADMIN_PASS.
* SSE Stream (stream.php -> Lines 2252-2346): Server-Sent Events open a 
  permanent streaming connection. The PHP script loops endlessly, tracking 
  the byte filesize() of modsecurity_audit.log. When it grows, it 
  parses the attacks natively and pushes the event instantly back to 
  the dashboard.
* The API System (api.php -> Lines 2016-2247): A custom backend switch 
  statement implementing an MVC model. Deploy websites natively, ban IPs 
  securely, and disable VirtualHosts remotely.

------------------------------------------------------------------------
 4. ARCHITECTURAL ASSESSMENT & HIGHLIGHTS
------------------------------------------------------------------------
1. EXTREME ROBUSTNESS IN AUTOMATION
   Creating OpenSSL certificates strictly using silent CLI flags and 
   formatting Apache VirtualHosts via bash heredocs eliminates thousands 
   of hours of manual labor usually required in Systems Admin classes.

2. REAL-TIME EVENT ARCHITECTURE
   Utilizing PHP as a buffer-less stream to parse and transport WAF logs 
   natively bridges the divide between backend engine auditing and 
   intuitive frontend Web GUI interfaces.

3. IMPENETRABLE SECURITY SEGREGATION
   By decoupling API privileges into a discrete sudoers whitelist, the 
   operating system ensures that even if the Control Panel were 
   compromised, hackers strictly cannot leverage it to execute arbitrary 
   malicious commands.
